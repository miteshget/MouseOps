A local web dashboard for managing multiple RHDP Showroom CI instances.
Stop clicking through individual Showroom tabs вЂ” run Solve and Validate across all your CIs from one place, with full persistence across page refreshes.

== Prerequisites

* https://git-scm.com[Git]
* https://www.python.org/downloads/[Python 3.11 or newer]
* https://nodejs.org[Node.js 18 or newer] (for building the React frontend)
* Network access to your Showroom instances (VPN if required)

Verify:

[source,bash]
----
python3 --version   # 3.11.x or higher
node --version      # 18.x or higher
----

== Installation

[source,bash]
----
git clone git@github.com:miteshget/MouseOps.git
cd MouseOps
----

== Starting the App

[source,bash]
----
bash run.sh
----

`run.sh` does everything automatically on first run:

. Builds the React frontend (`frontend/` в†’ `static/`)
. Creates a Python virtualenv
. Installs all Python dependencies
. Starts the FastAPI backend

----
MouseOps в†’ http://127.0.0.1:8765
----

Open http://127.0.0.1:8765 in your browser.
Stop the server with `Ctrl+C`.

=== Rebuilding the frontend after UI changes

[source,bash]
----
bash build.sh       # re-runs npm install + vite build
bash run.sh         # restart server (set SKIP_BUILD=1 to skip build step)
----

[source,bash]
----
SKIP_BUILD=1 bash run.sh    # skip build, start server immediately
----

=== Frontend development mode (hot reload)

Run the FastAPI backend and the Vite dev server side by side:

[source,bash]
----
# Terminal 1 вЂ” backend
SKIP_BUILD=1 bash run.sh

# Terminal 2 вЂ” frontend dev server with hot reload
cd frontend
npm install        # first time only
npm run dev        # http://127.0.0.1:5173 (proxies /api to :8765)
----

Runs a range of modules in order on the *server* вЂ” the loop survives page refreshes.

[cols="1,3"]
|===

| *в–¶ Validate*

| *в–¶в–¶ Both*

===== On Failure

When a module fails the server pauses and the card shows three options:

[cols="1,3"]
|===
| Button | Behaviour

| *рџ”„ Rerun*
| Re-runs the failed stage on the same module. Repeats until it passes or you stop.

| *вЏ­ Skip*
| Marks the module failed and moves to the next one.

| *в–  Stop*
| Stops the sequential run immediately.
|===

===== Page Refresh During a Sequential Run

The server owns the loop вЂ” it keeps running regardless of what the browser does.
On refresh, each tile reconnects automatically:

* If a sequential run is active, the tile locks its controls and `useSeqWatch` reconnects to the server loop.
* If an individual stream is active, the tile replays the buffered output and continues live.

A toast notification confirms the reconnect.

== Adding a CI Instance

. Click *пј‹ Add CI* (header or sidebar).
. Fill in the form:

[cols="1,3"]
|===
| Field | What to enter

| *Name*
| A short label, e.g. `LB1237 - RHEL10 - h9lhq`

| *Showroom URL*
| The base URL, e.g. `https://showroom.apps.cluster.example.com`

| *Auth Token*
| Bearer token if authentication is required. Leave blank if not needed.
  Leave blank when editing to keep the existing token.

| *Modules*
| Comma-separated list. Use *Auto-fill* instead of typing.
|===

=== Auto-fill Modules

. Enter the start number in *from* (use `0` for `module-00`).
. Enter the end number in *to*.
. Click *Fill*.

.Example
`from 0 to 7` в†’ `module-00, module-01, вЂ¦ module-07`

NOTE: Some labs start at `module-00`, others at `module-01`.

== Where Data Is Stored

[cols="1,3"]
|===
| Data | Location

| CI configurations (name, URL, token, modules)
| `cis.json` in the project root вЂ” created automatically.

| Active stream buffers and sequential run state
| Server memory вЂ” lost if the server restarts.

| Module run history (pass/fail per module)
| Browser `localStorage` вЂ” survives page refresh until browser data is cleared.

| Persisted log files (full Ansible output)
| `logs/{ci_id}/{module}/{stage}.log` in the project root.

| Tile visibility and order
| Browser `localStorage` under key `mouseops-visibility`.
|===

IMPORTANT: `cis.json` and `logs/` are in `.gitignore` and are never committed.
They may contain auth tokens and internal Showroom URLs.

== Troubleshooting

[cols="1,3"]
|===
| Problem | Fix

| *вќЊ Showroom unreachable вЂ” HTTP 404*
| The URL or module name is wrong. Open the CI in *вњЋ Edit* and verify the Showroom URL and module names.

| *вќЊ Cannot connect to Showroom*
| Showroom is down or unreachable. Check VPN and try the URL directly in your browser.

| *Sequential run lost after server restart*
| The server-side run state is in memory. If the server restarts, active runs are lost. Re-run from the card.

| *Port 8765 already in use*
| Stop the other process or change `--port 8765` in `run.sh`.

| *Frontend looks outdated after git pull*
| Run `bash build.sh` to rebuild, then restart the server.

| *Module summary missing after browser clear*
| `localStorage` was wiped. Re-run the modules to rebuild the table.

| *`python3: command not found`*
| Install Python 3.11+ and ensure it is on `$PATH`.

| *`node: command not found`*
| Install Node.js 18+ from https://nodejs.org.

| *Permission denied on run.sh / build.sh*
| `chmod +x run.sh build.sh`
|===
