= MouseOps
Mitesh, The Mouse <mitsharm@redhat.com>
:toc: left
:toc-title: Contents
:icons: font

A local web dashboard for managing multiple RHDP Showroom CI instances.
Stop clicking through individual Showroom tabs — run Solve and Validate across all your CIs from one place, with full persistence across page refreshes.

== Prerequisites

* https://git-scm.com[Git]
* https://www.python.org/downloads/[Python 3.11 or newer]
* Network access to your Showroom instances (VPN if required)

[source,bash]
----
python3 --version   # must show 3.11.x or higher
----

== Installation

[source,bash]
----
git clone git@github.com:miteshget/MouseOps.git
cd MouseOps
----

== Starting the App

[source,bash]
----
bash run.sh
----

First run creates a virtualenv and installs dependencies automatically.
Every subsequent run starts immediately.

----
MouseOps → http://127.0.0.1:8765
----

Open http://127.0.0.1:8765. Stop with `Ctrl+C`.

== Adding a CI Instance

. Click *＋ Add CI* in the top-right corner.
. Fill in the form:
+
[cols="1,3"]
|===
| Field | What to enter

| *Name*
| A short label, e.g. `LB1237 - RHEL10 - h9lhq`

| *Showroom URL*
| The base URL, e.g. `https://showroom.apps.cluster.example.com`

| *Auth Token*
| Bearer token if authentication is required. Leave blank if not needed.
Leave blank when editing to keep the existing token.

| *Modules*
| Comma-separated list, e.g. `module-01,module-02,module-03`.
Use *Auto-fill* instead of typing (see below).
|===

. Click *Add CI*.

=== Auto-fill Modules

. Enter the start number in *from* (use `0` for `module-00`).
. Enter the last number in *to*.
. Click *Fill*.

.Example
`from 0 to 7` → `module-00, module-01, … module-07`

NOTE: Some labs start at `module-00`, others at `module-01`.

== CI Card Layout

Each card has two sections.

=== Individual Execution

Runs one action on one selected module.

. Pick the module from the *Module* dropdown.
. Click:

[cols="1,3"]
|===
| Button | What it does

| *🚀 Solve*
| Runs the solve playbook on the selected module.

| *✓ Validate*
| Runs the validate playbook on the selected module.
|===

Live task chips appear as the playbook runs.
Expand *Show logs* for full Ansible output, or click *⤢* to open it full-screen.

=== Sequential Execution

Runs a range of modules in order on the server — persists across page refreshes.

. Set *Range* using the *from* and *to* dropdowns.
. Click:

[cols="1,3"]
|===
| Button | What it does

| *▶ Solve*
| Runs solve on every module in the range, in order.

| *▶ Validate*
| Runs validate on every module in the range, in order.

| *▶▶ Both*
| Runs solve then validate on each module before moving to the next.
Only continues to the next module when the current one passes both steps.
|===

==== On Failure

When a module fails the run pauses and shows three options:

[cols="1,3"]
|===
| Button | Behaviour

| *🔄 Rerun*
| Re-runs the failed stage on the same module. Repeats until it passes or you stop.

| *⏭ Skip & Continue*
| Marks the module failed and moves to the next one.

| *■ Stop*
| Stops the sequential run immediately.
|===

==== Page Refresh During a Sequential Run

The sequential loop runs on the server, not in the browser.
Refreshing the page reconnects automatically — the run continues from exactly where it was.
A toast notification confirms the reconnect.

==== Cancel at Any Time

Click *■ Stop* to cancel the current stream or sequential run.

== Bulk Actions (All CIs)

Click *Actions ▾* in the top-right corner to act on every CI at once.

[cols="1,1,3"]
|===
| Section | Item | Behaviour

| Individual
| 🚀 Solve All
| Fires solve on every card's currently selected module simultaneously.

| Individual
| ✓ Validate All
| Fires validate on every card's currently selected module simultaneously.

| Sequential
| ▶ Solve All
| Runs sequential solve on each CI, one CI at a time.

| Sequential
| ▶ Validate All
| Runs sequential validate on each CI, one CI at a time.

| Sequential
| ▶▶ Run All
| Runs sequential solve + validate on each CI, one CI at a time.

| Config
| ⬇ Export Config
| Downloads all CI configurations as a JSON file (no IDs — safe to import on another machine).

| Config
| ⬆ Import Config
| Opens a file picker to load a previously exported JSON config.

| Results
| 🗑 Clear All Results
| Wipes the module summary table for every CI (with confirmation).
|===

NOTE: Sequential bulk actions process CIs one after another — the next CI does not start until the previous one finishes.

== Module Summary Table

After any action a summary table appears on the card:

[cols="1,1,1"]
|===
| Module | Solve | Validate

| module-01
| ✅ Pass 📄
| ✅ Pass 📄

| module-02
| ❌ Fail 📄
| —

| module-03
| 🔵 running…
| —
|===

* Appears automatically — no action needed.
* Updates live as each module completes.
* Persists across page refreshes via browser `localStorage`.
* Re-running a module overwrites its row.
* Click *📄* next to any badge to open that module's persisted log in the full-screen viewer.
* Rows for modules that have been removed from the CI config are greyed out with ⚠.

== Card Header Buttons

[cols="1,3"]
|===
| Button | Action

| *📄*
| Load the last run's log into the full-screen viewer.

| *🗑*
| Clear this CI's module summary results (with confirmation). Other CIs are not affected.

| *✎*
| Edit the CI's name, URL, token, or module list.

| *✕*
| Remove the CI and clear its history (with confirmation).
|===

== Log Viewer

* Inline logs appear in a collapsible *Show logs* panel at the bottom of each card.
* Click *⤢* in the log bar to open a full-screen dark terminal view.
* If a run is active, new lines appear in real time in the full-screen view.
* Click *📄* in the module summary table to load any module's persisted log directly.

== Editing and Removing CIs

* *✎ Edit* — update the name, URL, token, or module list.
* *✕ Remove* — deletes the CI and its history (with confirmation).
* Editing does not interrupt a running stream.

TIP: Leave *Auth Token* blank when editing to keep the existing token unchanged.

== Where Data Is Stored

[cols="1,3"]
|===
| Data | Location

| CI configurations (name, URL, token, modules)
| `cis.json` in the project directory — created automatically.

| Active stream buffers and sequential run state
| Server memory — lost if the server restarts.

| Module run history (pass/fail per module)
| Browser `localStorage` — survives page refresh until browser data is cleared.

| Persisted logs (full Ansible output)
| `logs/{ci_id}/{module}/{stage}.log` in the project directory.
|===

IMPORTANT: `cis.json` and `logs/` are in `.gitignore` and are never committed.
They may contain auth tokens and internal Showroom URLs.

== Troubleshooting

[cols="1,3"]
|===
| Problem | Fix

| *❌ Showroom unreachable — HTTP 404*
| The URL or module name is wrong. Open the CI in *✎ Edit* and verify the Showroom URL and module names.

| *❌ Cannot connect to Showroom*
| Showroom is down or unreachable. Check VPN and try opening the URL directly in your browser.

| *Sequential run lost after server restart*
| The server-side run state is in memory. If the server restarts (e.g. uvicorn `--reload` auto-reload), active runs are lost. Re-run from the card.

| *Port 8765 already in use*
| Stop the other process or edit `run.sh` to change `--port 8765`.

| *Module summary missing after browser clear*
| `localStorage` was wiped. Re-run the modules to rebuild the table.

| *`python3: command not found`*
| Install Python 3.11+ and ensure it is on `$PATH`.

| *Permission denied on run.sh*
| `chmod +x run.sh`
|===
