= MouseOps
Mitesh, The Mouse <mitsharm@redhat.com>
:toc: left
:toc-title: Contents
:icons: font

A local web dashboard for managing multiple RHDP Showroom instances.
Run Solve and Validate across all your Showrooms from one place, with full persistence across page refreshes and server restarts.

== Prerequisites

* https://git-scm.com[Git]
* https://www.python.org/downloads/[Python 3.11 or newer]
* https://nodejs.org[Node.js 18 or newer] (for building the React frontend)
* Network access to your Showroom instances (VPN if required)

[source,bash]
----
python3 --version   # 3.11.x or higher
node --version      # 18.x or higher
----

== Installation

[source,bash]
----
git clone git@github.com:miteshget/MouseOps.git
cd MouseOps
----

== Starting the App

`run.sh` handles everything automatically: builds the React frontend, creates a Python virtualenv, installs dependencies, generates a TLS certificate, and starts the server.

=== HTTPS mode (default — recommended)

[source,bash]
----
bash run.sh
# or explicitly:
bash run.sh --https
----

----
MouseOps  [HTTPS mode]
  HTTP  → http://127.0.0.1:8765  (redirects to HTTPS)
  HTTPS → https://127.0.0.1:8766
  Note: browser will warn about self-signed cert — click Advanced → Proceed
----

A self-signed TLS certificate is generated automatically in `.ssl/` on first run.
It is valid for 10 years and covers `localhost` / `127.0.0.1`.

=== HTTP mode (no SSL)

[source,bash]
----
bash run.sh --http
----

----
MouseOps  [HTTP mode]
  http://127.0.0.1:8765
----

=== Custom ports

[source,bash]
----
MOUSEOPS_HTTP_PORT=80 MOUSEOPS_HTTPS_PORT=443 bash run.sh --https
MOUSEOPS_HTTP_PORT=8080 bash run.sh --http
----

=== Other options

[source,bash]
----
SKIP_BUILD=1 bash run.sh        # skip React build step
bash build.sh                   # rebuild frontend only
----

=== Frontend development mode (hot reload)

[source,bash]
----
# Terminal 1 — backend (HTTP mode is simpler for dev)
SKIP_BUILD=1 bash run.sh --http

# Terminal 2 — Vite dev server
cd frontend
npm install   # first time only
MOUSEOPS_DEV_TARGET=http://127.0.0.1:8765 npm run dev
# opens http://127.0.0.1:5173 and proxies /api to the backend
----

Stop the server with `Ctrl+C`.

== Authentication

Every user must sign in before accessing the dashboard.

=== Default credentials

----
Username: admin
Password: mouseops
----

Change the admin password immediately after first login.

=== User roles

[cols="1,1,3"]
|===
| Role | Who | What they can do

| *admin*
| Lab engineers
| Full access: run Solve/Validate, manage CIs, manage users, clear results, export/import config.

| *viewer*
| Stakeholders, observers
| Read-only: view tiles, watch live streams, load logs. Cannot trigger any actions.
|===

=== Managing users (admin only)

. Click your username in the top-right header.
. Select the *Users* tab.

From there you can:

* *Create* a new user with `admin` or `viewer` role.
* *Toggle role* by clicking the role badge next to a username.
* *Remove* a user (cannot remove yourself or the last admin).

=== Changing your password

. Click your username in the top-right header.
. Select the *Change Password* tab.

All users can change their own password at any time.

=== Sign out

Click *Sign out* in the top-right header.
Signing out shows the login overlay — all running streams and sequential runs continue uninterrupted in the background.

=== Emergency read-only lockdown

Set `MOUSEOPS_READONLY=1` to block all write operations server-side, even for admin accounts:

[source,bash]
----
MOUSEOPS_READONLY=1 bash run.sh
----

A *Read-only* badge appears in the header.

== UI Overview

=== Left Sidebar

The collapsible sidebar lists every Showroom instance.

[cols="1,3"]
|===
| Control | Purpose

| Toggle switch
| Show or hide a tile on the main dashboard.

| *All* / *None* buttons
| Show or hide every tile at once.

| Search box
| Filter the list by name or URL in real time.

| Drag handle (⠿)
| Drag tiles to reorder them. Order is saved across refreshes.

| *＋ Add Showroom* button
| Open the Add Showroom form (admin only).
|===

=== Main Dashboard

The dashboard shows only the tiles you have toggled on.
Each Showroom is a card with two sections.

==== Individual Execution

Runs one action on one selected module.

. Pick the target module from the *Module* dropdown.
. Click an action:

[cols="1,3"]
|===
| Button | What it does

| *🚀 Solve*
| Runs the solve playbook on the selected module.

| *✓ Validate*
| Runs the validate playbook on the selected module.
|===

Live task chips appear as the playbook runs.
Expand *Show logs* for full Ansible output, or click *⤢* to open it full-screen.

==== Sequential Execution

Runs a range of modules in order — the loop runs on the *server* and survives both page refreshes and server restarts.

. Set *Range* using the *from* and *to* dropdowns.
. Click an action:

[cols="1,3"]
|===
| Button | What it does

| *▶ Solve*
| Runs solve on every module in the range, in order.

| *▶ Validate*
| Runs validate on every module in the range, in order.

| *▶▶ Both*
| Runs solve then validate on each module before moving to the next.
  Only moves forward when the current step passes.
|===

===== On Failure

When a module fails the server pauses and the card shows three options:

[cols="1,3"]
|===
| Button | Behaviour

| *🔄 Rerun*
| Re-runs the failed stage on the same module. Repeats until it passes or you stop.

| *⏭ Skip*
| Marks the module failed and moves to the next one.

| *■ Stop*
| Stops the sequential run immediately.
|===

===== Surviving Page Refresh and Server Restart

The sequential loop runs on the server as a background task:

* *Page refresh* — the browser reconnects automatically. A toast confirms the reconnect.
* *Server restart* — progress is saved to `seq_state.json` before each module. On restart, the loop resumes from where it stopped.

=== Module Summary Table

After any action a summary table appears on the card:

[cols="1,1,1"]
|===
| Module | Solve | Validate

| module-01
| ✅ Pass 📄
| ✅ Pass 📄

| module-02
| ⟳ running
| —

| module-03
| ❌ Fail 📄
| —
|===

* Appears automatically — no action needed.
* Shows a live spinner while a module is running.
* Updates in real time as each module completes.
* Persists across page refreshes via browser `localStorage`.
* Re-running a module overwrites its row.
* Click *📄* next to any badge to open that module's persisted log.
* Rows for modules removed from the CI config are greyed out with ⚠.

[cols="1,3"]
|===
| Button | Action

| *📄*

| *🗑*

== Adding a Showroom Instance

. Click *＋ Add Showroom* (header or sidebar) — admin only.
. Fill in the form:

[cols="1,3"]
|===
| Field | What to enter

| *Name*
| A short label, e.g. `LB1237 - RHEL10 - h9lhq`

| *Showroom URL*
| The base URL, e.g. `https://showroom.apps.cluster.example.com`

| *Auth Token*
| Bearer token if authentication is required. Leave blank if not needed.
  Leave blank when editing to keep the existing token.

| *Modules*
| Comma-separated list. Use *Auto-fill* instead of typing.
|===

=== Auto-fill Modules

. Enter the start number in *from* (use `0` for `module-00`).
. Enter the end number in *to*.
. Click *Fill*.

.Example
`from 0 to 7` → `module-00, module-01, … module-07`

NOTE: Some labs start at `module-00`, others at `module-01`.

== Bulk Actions (All Showrooms)

Click *Actions ▾* in the top-right corner (admin only) to act on every Showroom at once.

[cols="1,1,3"]
|===
| Section | Item | Behaviour

| Sequential
| ▶ Solve All
| Starts sequential solve-only on each Showroom, one at a time.

| Sequential
| ▶ Validate All
| Starts sequential validate-only on each Showroom, one at a time.

| Sequential
| ▶▶ Run All
| Starts sequential solve + validate on each Showroom, one at a time.

| Config
| ⬇ Export Config
| Downloads all Showroom configs as a JSON file (tokens excluded from response — re-enter after import).

| Config
| ⬆ Import Config
| Loads a previously exported JSON file and adds the Showrooms.

| Results
| 🗑 Clear All Results
| Wipes the module summary for every Showroom (with confirmation).
|===

== Where Data Is Stored

[cols="1,3"]
|===
| Data | Location

| Showroom configs (name, URL, encrypted token, modules)
| `cis.json` — tokens are encrypted with AES-128 (Fernet); the encryption key is in `.secret.key`.

| Users and hashed passwords
| `users.json` — passwords are SHA-256 hashed with a per-user salt; never stored in plaintext.

| TLS certificate and key
| `.ssl/cert.pem` and `.ssl/key.pem` — auto-generated self-signed cert, valid 10 years.

| Active stream buffers
| Server memory — reconnected automatically by the browser on page refresh.

| Sequential run progress
| `seq_state.json` — saved before each module so runs resume after a server restart.

| Module run history (pass/fail)
| Browser `localStorage` — survives page refresh until browser data is cleared.

| Persisted log files (full Ansible output)
| `logs/{showroom_id}/{module}/{stage}.log`

| Tile visibility and order
| Browser `localStorage` under key `mouseops-visibility`.
|===

IMPORTANT: `cis.json`, `users.json`, `.secret.key`, `.ssl/`, `logs/`, and `seq_state.json` are all in `.gitignore` and are never committed.

== Security Notes

* Showroom auth tokens are encrypted at rest with AES-128 (Fernet). The encryption key (`.secret.key`) must be kept safe — if lost, tokens must be re-entered.
* Session cookies are `HttpOnly` and `SameSite=Strict`. In HTTPS mode they also carry the `Secure` flag so they are never sent over plain HTTP.
* The self-signed TLS certificate will trigger a browser warning. Click *Advanced → Proceed* to continue. For production use, replace `.ssl/cert.pem` and `.ssl/key.pem` with a certificate from a trusted CA and restart.

== Troubleshooting

[cols="1,3"]
|===
| Problem | Fix

| *Browser shows self-signed cert warning*
| Click *Advanced → Proceed* (Chrome/Edge) or *Accept the Risk* (Firefox).
  For production, replace `.ssl/cert.pem` and `.ssl/key.pem` with a trusted cert.

| *Forgot admin password*
| Delete `users.json` and restart. The default `admin / mouseops` account is recreated automatically.

| *`.secret.key` lost or deleted*
| Regenerates on next start, but all existing Showroom tokens will fail to decrypt.
  Open each Showroom in *✎ Edit* and re-enter its token.

| *❌ Showroom unreachable — HTTP 404*
| URL or module name is wrong. Open the Showroom in *✎ Edit* and verify.

| *❌ Cannot connect to Showroom*
| Showroom is down or unreachable from your network. Check VPN and try the URL in your browser.

| *Sequential run not resumed after restart*
| Check that `seq_state.json` was not manually deleted. If missing, restart the run from the card.

| *Port already in use*
| Change ports: `MOUSEOPS_HTTP_PORT=8080 MOUSEOPS_HTTPS_PORT=8443 bash run.sh`

| *Frontend looks outdated after `git pull`*
| Run `bash build.sh` to rebuild, then restart.

| *Module summary missing after browser clear*
| `localStorage` was wiped. Re-run the modules to rebuild the table.

| *`python3: command not found`*
| Install Python 3.11+ from https://www.python.org and add it to `$PATH`.

| *`node: command not found`*
| Install Node.js 18+ from https://nodejs.org.

| *Permission denied on run.sh / build.sh*
| `chmod +x run.sh build.sh`
|===
