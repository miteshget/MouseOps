Mitesh, The Mouse <mitsharm@redhat.com>
:toc: left
:toc-title: Contents
:icons: font

A local web dashboard for managing multiple RHDP Showroom CI instances.
Stop clicking through individual Showroom tabs — run Solve and Validate across all your CIs from one place.

== Prerequisites

Before you begin, make sure you have:

* https://git-scm.com[Git]
* https://www.python.org/downloads/[Python 3.11 or newer]
* Network access to your Showroom instances (VPN if required)

Verify Python is ready:

[source,bash]
----
python3 --version
----

You should see `Python 3.11.x` or higher.

== Installation

Clone the repository and enter the project directory:

[source,bash]
----
git clone git@github.com:miteshget/MouseOps.git
cd MouseOps
----

== Starting the App

[source,bash]
----
bash run.sh
----

The first run automatically creates a Python virtualenv and installs all dependencies.
Every subsequent run skips setup and starts the server immediately.

You should see:

----
MouseOps → http://127.0.0.1:8765
----

Open http://127.0.0.1:8765 in your browser.

To stop the server press `Ctrl+C`.

== First Steps

When you open MouseOps for the first time the dashboard is empty.
Start by adding the CI instances you want to manage.

=== Add a CI Instance

. Click *+ Add CI* in the top-right corner.
. Fill in the form:
+
[cols="1,3"]
|===
| Field | What to enter

| *Name*
| A short label so you can tell CIs apart, e.g. `LB1237 - RHEL10 - h9lhq`

| *Showroom URL*
| The base URL of the Showroom, e.g. `https://showroom.apps.cluster.example.com`

| *Auth Token*
| Bearer token if authentication is required. Leave blank if not needed.

| *Modules*
| The modules that exist on this Showroom. Type them comma-separated or use *Auto-fill* (see below).
|===

. Click *Add CI*. The card appears on the dashboard immediately.

Repeat for every CI you want to manage.

=== Auto-fill Modules

If your lab has a numbered sequence of modules, use the *Auto-fill* helper instead of typing each one:

. In the *from* field enter the first module number (use `0` if your lab starts at `module-00`).
. In the *to* field enter the last module number.
. Click *Fill*.

The modules field fills automatically with zero-padded names.

.Example
`from 0 to 7` → `module-00, module-01, module-02, ... module-07`

NOTE: Some labs start at `module-00`, others at `module-01`. Check your Showroom page before filling.

== Running Solve and Validate

Each CI card has its own controls.

=== Single Module

. Pick the module from the *Module* dropdown at the top of the card.
. Click the action you want:

[cols="1,3"]
|===
| Button | What it does

| *🚀 Solve*
| Runs the solve playbook on the selected module. Use this to automatically complete a lab step.

| *✓ Validate*
| Runs the validate playbook on the selected module. Use this to check whether a step is correctly completed.
|===

Live task progress appears on the card as coloured chips.
Expand *Show logs* to see the full Ansible output.

=== Bulk Actions

To trigger the same action across all CIs at once, use the header buttons:

* *🚀 Solve All* — runs Solve on every card's currently selected module simultaneously.
* *✓ Validate All* — runs Validate on every card's currently selected module simultaneously.

== Sequential Run

*▶▶ Run All* runs every module on a card in order — Solve first, then Validate — only moving to the next module when the current one passes both steps.

=== Run a Range of Modules

Use the *from* and *to* dropdowns next to *▶▶ Run All* to limit which modules are included:

.Example
Set `from module-03 to module-10` to start at module-03 and stop after module-10, skipping everything outside that range.

The *to* dropdown defaults to the last module so the default behaviour is to run everything.

=== What Happens When a Module Fails

MouseOps pauses and shows two options:

[cols="1,3"]
|===
| Button | Behaviour

| *⏭ Skip & Continue*
| Skips the failed module and continues with the next one.
The failed module stays marked ❌ in the summary table.

| *■ Stop*
| Stops the run immediately.
|===

=== Cancel at Any Time

Click *■ Stop* to cancel the currently running stream or sequential run.

== Module Summary Table

After any action, a summary table appears on the card:

[cols="1,1,1"]
|===
| Module | Solve | Validate

| module-01
| ✅ Pass
| ✅ Pass

| module-02
| ❌ Fail
| —

| module-03
| —
| —
|===

* The table appears automatically after the first run — no action needed.
* It updates in real-time as each module completes.
* It persists across page refreshes — your history is not lost when you reload.
* Re-running a module overwrites its row with the latest result.

== Editing and Removing CIs

* Click *✎* on a card to edit the name, URL, token, or module list.
* Click *✕* to remove the CI and clear its history.

TIP: When editing, leave the *Auth Token* field blank to keep the existing token unchanged.
Only fill it in when you want to set a new one.

== Where Data Is Stored

[cols="1,3"]
|===
| Data | Where

| CI configurations (name, URL, token, modules)
| `cis.json` in the project directory — created automatically on first use.

| Module run history (pass/fail per module)
| Browser `localStorage` — persists until you clear browser data.
|===

IMPORTANT: `cis.json` is in `.gitignore` and will never be committed.
It is local to your machine and may contain auth tokens and internal Showroom URLs.

== Updating

Pull the latest changes and restart:

[source,bash]
----
git pull
bash run.sh
----

No manual dependency updates are needed — `run.sh` keeps the virtualenv in sync.

== Troubleshooting

[cols="1,3"]
|===
| Problem | Fix

| *❌ Showroom unreachable — HTTP 404*
| The URL or module name is wrong. Open the CI in *Edit* and double-check the Showroom URL.
  Also verify the module name exists on that Showroom.

| *❌ Cannot connect to Showroom*
| The Showroom is down or not reachable from your network. Check your VPN connection and try opening the Showroom URL in your browser directly.

| *Port 8765 already in use*
| Another process is using the port. Stop it or edit `run.sh` to change `--port 8765` to a free port.

| *Module summary missing after browser clear*
| `localStorage` was wiped. Re-run the affected modules to rebuild the summary table.

| *`python3: command not found`*
| Install Python 3.11+ from https://www.python.org/downloads/ and ensure it is on your `PATH`.

| *Permission denied on run.sh*
| Make it executable: `chmod +x run.sh`
|===
